# デバッグ：大容量ファイル読み込み失敗

## 判明した新事実 (Phase 4)

### 1. 「Windowsで[Strategy]ログが出ない」件について
- **原因**: ユーザーご提示のログ (`app.js:1146 [libraryLoad] openEpub`) は、新規にファイルを追加する `handleFile` 関数ではなく、**ライブラリに保存済みの既存の本を開く `openFromLibrary` 関数**を通ったときのものです。
- **仕様**: `[Strategy]` などの戦略ログは**新規ファイル追加時(`handleFile`)のみ**出力される仕様でした（これは正常な動作です）。コードは（Service Worker更新により）正しく動作しており、古いキャッシュの問題は解決しています。

### 2. 「Androidで400MBのZIPが開けず（クラッシュ）」の件について
- **原因 (JSZipのメモリ設計)**: ZIPファイルを解凍するライブラリ「JSZip」は、内部的に**ファイル全体をメモリ(ArrayBufferやUint8Array)に読み込んで処理する仕様**になっています。ストリーミング展開（分割処理）に対応していません。
- **OOM(メモリ枯渇)**: 400MBのZIPファイルを `JSZip.loadAsync()` に渡すと、内部確保領域などを合わせて **約1GBの連続したRAM** を要求します。
- **結論**: Android等モバイルブラウザの仕様で1つのタブが使えるメモリは500MB〜1GB程度に制限されているため、**400MBのZIPファイル(CBZ)はJSZipの設計限界・ブラウザの限界を超えており、物理的に開くことができません（ブラウザ自体がクラッシュします）。**

## 今回の修正措置
- [x] `archive-handler.js` にて、**ZIP形式(CBZ)にも「クラッシュ保護（メモリ制限エラー）」を導入**。
- ※ モバイル端末(Android等)の場合は **100MB**、PC等の場合は **300MB** を超える巨大ZIPは、強制終了(OOM)を未然に防ぎ「事前解凍またはEPUB化」を案内するエラーメッセージを表示するよう改修しました。
- ※ Windowsで1GB級RAMが確保できる環境なら400MBでも偶然開ける場合がありますが、非常に不安定です。
